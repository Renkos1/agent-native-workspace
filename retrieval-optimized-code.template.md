# Retrieval-optimized code — addressability rules (the "vectorization" ruling)

Ruling: **no codebase embeddings, no vector index** — stale the moment code moves, adds
infrastructure, and doesn't improve edit precision. The actual goal (a model jumping
O(1) to any concept and modifying it, zero exploratory grep) is achieved with
**addressability rules**:

1. **Canonical vocabulary**: identifiers, filenames, routes, and log strings use glossary
   terms exactly (`{{glossary path}}` is the authority). One concept, one word,
   repo-wide; synonyms and off-glossary abbreviations banned. A model that knows the
   glossary constructs the right search string on the first try — that is cheap,
   never-stale "semantic search."
2. **The generated index is the "vector store"**: `MAP.md` (module → responsibility →
   key files) + generated tables (route table / command table, generated from
   `contracts/`). **Regenerated by script, never hand-maintained** — a stale index is
   worse than none. (Implementation hint: structural freshness comes from scanning; the
   one-line responsibility strings live in the generator's SEED table; an unregistered
   new directory gets flagged so its row gets added.) Agent rule: read the index, then
   jump; whole-tree exploration is a smell worth reporting.
3. **One concept, one file**; size caps {{per tier, e.g. new web code ≤400 lines}}. A
   filename must predict its contents.
4. **Stable anchors**: exported symbols renamed conservatively; docs and briefs
   reference `path:symbol`, never line numbers (line numbers rot faster than symbols).
5. **Colocation**: tests sit beside their sources (or centralized — but declared in
   MAP); contract consumers are named after the contract.
6. **Zero dead code, zero duplicate members** (lint-enforced). Dead code poisons
   retrieval worse than it poisons runtime — a model pays real cost discovering which
   hundreds of lines are alive.

Net effect: vocabulary → index → file → symbol. Three hops, no scanning.
