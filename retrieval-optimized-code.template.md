# Retrieval-optimized code — 可寻址性规则（「矢量化」裁决）

裁决: **不建代码库 embedding/向量索引**——代码一动就过期、添基建、不提升编辑精度。
真实目标（模型 O(1) 跳定位并修改任意概念、零探索性 grep）经**可寻址性规则**达成:

1. **Canonical 词汇**: 标识符/文件名/路由/日志串精确使用词汇表术语（`{{glossary 路径}}`
   是词汇权威）。一概念一词, 全仓禁同义词、禁表外缩写。知道词汇表的模型第一次就能构造
   出正确搜索串——这就是廉价、永不过期的「语义搜索」。
2. **生成式索引是「向量库」**: `MAP.md`（模块 → 职责 → 关键文件）+ 生成表（路由表/
   命令表, 从 `contracts/` 生成）。**脚本再生, 永不手维护**——过期索引比没有更糟
   （实现提示: 结构由扫描保证新鲜, 一行职责文案维护在生成脚本的 SEED 表里, 扫到表外
   新目录即标「未登记」提醒补行）。Agent 规则: 读索引再跳转; 全树探索是要上报的坏味道。
3. **一概念一文件**; 尺寸上限 {{按层定, 示例: web 新代码 ≤400 行}}。文件名必须能预测内容。
4. **稳定锚点**: 导出符号重命名稳定; 文档/brief 引用 `path:symbol` 而非行号
   （行号比符号腐烂快）。
5. **共置**: 测试贴源文件（或集中式但在 MAP 声明）; 契约的消费者按契约命名。
6. **零死码、零重复成员**（lint 强制）。死码毒害检索甚于毒害运行时——模型会花真实
   成本去发现哪几百行是活的。

效果: 词汇 → 索引 → 文件 → 符号, 三跳无扫描。
